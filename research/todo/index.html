    Schedule (Medium Impact)¶

    • ~~Carry trade strategy~~ — DONE (enabled, in paper-trade validation): CarryManager in src/forex_bot/strategy/carry.py. Fetches interest rates from FRED (with config fallbacks), scores pairs by differential, weekly (Monday 14:00 UTC) rebalances through existing ExecutionEngine/RiskManager pipeline. Strategy-aware risk routing (separate carry/straddle rule sets). Default basket: USDZAR, USDTRY, USDMXN, AUDJPY, NZDJPY. Now carry.enabled: true — monitor a few clean weekly rebalances before trusting it live. (Rebalance timezone + Sunday→Monday schedule fixed in PR #55.) 23 unit tests. PR #44.
    • ~~Currency momentum strategy~~ — REMOVED (2026-07-05): built as time-series momentum, validated, then removed as carry-redundant with no diversifying edge. The strategy code (strategy/momentum.py), config, risk routing, scheduling, and tests were deleted; the analysis is retained in 13-mc-momentum.md + scripts/mc_momentum.py as the record of why. MC validation (13-mc-momentum.md, scripts/mc_momentum.py): VERDICT = BORDERLINE (paper-trade only) — strong in-sample (Sharpe 1.14, +13%/yr on 2020-2024) but the edge largely decays out-of-sample (Sharpe 0.14, +0.7%/yr on 2025-2026; MC 5th-pct −8.4%, 44% chance of a losing OOS period). Best params: lookback 6m, min_return 5%. Do NOT enable live on this evidence. Expanded-universe reevaluation (17 pairs incl. straddle-failures GBPJPY/CADJPY/EURCAD/GBPCAD): the OOS momentum edge is concentrated almost entirely in USDTRY (OOS Sharpe 5.91) — only 3/17 pairs pass OOS, broadening the basket makes it worse (17-pair OOS Sharpe −0.09), and USDTRY's trend overlaps the carry book (same lira-depreciation move), so broad momentum is largely redundant with carry. Straddle-failed crosses GBPJPY/EURCAD are only weak-positive under momentum. Next steps: (1) accumulate paper results, compare live vs backtest via /trade-review; (2) if pursued, treat momentum as a small USDTRY-centric sleeve with concentration limits, not a diversified book — or drop it as carry-redundant; (3) consider a cross-sectional variant and a rolling multi-fold walk-forward with cost sensitivity. Background: Sharpe ~0.95, uncorrelated with carry (BIS WP 366, ScienceDirect, Kellogg/Northwestern).
    • Evaluate strategy-validation methods beyond MC walk-forward — the current pipeline relies on Monte Carlo + train/test walk-forward + Bonferroni-adjusted CIs. Research additional/complementary methods to harden strategy evaluation and catch overfitting, especially for the newer strategies (carry, momentum) that lack the straddle's event-level MC rigor: Deflated Sharpe Ratio and Probability of Backtest Overfitting (PBO) (López de Prado); Combinatorial Purged Cross-Validation (CPCV); White's Reality Check / Hansen's SPA test for data-snooping across the many pair/param combos already tried; block/stationary bootstrap of returns for CI robustness; regime-conditioned backtests (crisis vs calm) and stress tests; and formalizing the live-vs-backtest drift tracking already partially done in reporting/weekly.py. Deliverable: pick 1-2 to add to the mc-analysis workflow and document in a research note.
    • Value / Purchasing Power Parity (PPP) strategy — SCAFFOLD BUILT (unvalidated, disabled): ValueManager in src/forex_bot/strategy/value.py. Uses relative PPP: builds each pair's real exchange rate (nominal price × relative CPI, CPI from FRED) and measures its deviation from the long-run mean as a z-score — long undervalued (z < 0), short overvalued (z > 0), ranked by |z|, capped at max-concurrent, monthly rebalance (1st, 14:37 UTC). Developed-market majors only (EURUSD, GBPUSD, USDJPY, USDCAD, AUDUSD, NZDUSD, USDCHF) — their real rates mean-revert, unlike exotics that trend, so value is genuinely uncorrelated with carry. Strategy-aware risk routing (_value_rules), sweep exclusion, 24 unit tests. Starts value.enabled: false. MC validation DONE (14-mc-value-ppp.md, scripts/mc_value.py): VERDICT = INCONCLUSIVE. In-sample is flat (Sharpe 0.03 over ~119 months) — no durable edge from 1-month PPP reversion on developed majors (consistent with PPP acting over years). The strong-looking OOS (Sharpe 1.29) is only 19 months = small-sample noise, not validation. The one real positive: value's returns are −0.47 correlated with a carry factor — a genuine diversifier (the whole rationale). Longer-horizon test DONE (holding-horizon sweep 1/3/6/12 months with non-overlapping information coefficients, in scripts/mc_value.py / report 14): lengthening the hold does NOT rescue it — the real-exchange-rate z-score has ~zero predictive power at every horizon (IC between −0.04 and +0.00, all |t| < 0.6, over 55-716 obs). The PPP signal simply has no edge on developed majors here. Conclusion: SHELVE (like momentum) — keep value.enabled: false; the only residual value is the −0.47 carry diversification, but with ~zero expected return it isn't tradeable. Both factor experiments (momentum, value) came back empty — the bot's edge remains the event straddle.
    • Statistical arbitrage / mean reversion — Find correlated currency pairs (e.g., AUDUSD vs NZDUSD) that normally move together. When the spread between them widens beyond a statistical threshold (e.g., 2 standard deviations), buy the cheap one and sell the expensive one. Close when they converge. Market-neutral — uncorrelated with directional strategies. Sharpe ~1.0+ when it works. Main risk: correlations break permanently (e.g., 2015 CHF de-peg). Implementation: identify cointegrated pairs, calculate z-scores, set entry/exit thresholds. Research needed: cointegration tests on IB forex pairs, regime-change detection to avoid broken correlations. Timeframe: days to weeks.

    Strategy Research Candidates (added 2026-07-17)¶

    Quant/fundamental strategies not yet on the roadmap, ordered by fit with existing data and infrastructure. Pure technical-analysis approaches (MA crossovers, chart patterns) and grid/martingale sizing schemes are deliberately excluded — no quantitative edge or unbounded risk. Triangular/latency arbitrage excluded as non-viable at retail latency.

    • ~~MC-validate the existing SurpriseStrategy (post-news directional drift)~~ — RESOLVED NEGATIVE (2026-07-21): docs/research/15-mc-surprise.md — all 36 grid cells fail (both surprise definitions); all 30 computable walk-forward OOS results negative; FOMC effectively untradeable (2/53 decisions had nonzero surprise). Verdict: remove surprise.py. Original item: strategy/surprise.py was already built (trades the direction of a data surprise post-release, surprise_threshold_pct: 10, TP/SL 25/15) but has never been MC/walk-forward validated — no research report covers it, unlike every other active strategy. The academic base is solid: news surprises produce directional drift that persists minutes-to-hours after release (Andersen, Bollerslev, Diebold & Vega, AER 2003; ECB WP on post-announcement FX drift). All required data is already in hand: 996+ events with forecast/actual in the events DB plus 1-min Dukascopy bars around each. Extend scripts/monte_carlo_dukascopy.py to simulate surprise-direction entries N seconds post-release across surprise-magnitude thresholds, with the same walk-forward split (train 2020-2024, test 2025-2026). Deliverable: research report 15; enable or delete the module based on the verdict. This is the highest-priority candidate — the code exists and is one bad config edit away from trading unvalidated.
    • Aggregate economic surprise index (nowcast tilt) — Citi Economic Surprise Index-style: aggregate standardized surprises (actual vs forecast, normalized by historical surprise dispersion) per currency over a rolling window; tilt long currencies with improving data momentum, short deteriorating ones, rebalance weekly. Documented predictive content for FX over days-to-weeks horizons. Unique fit: the events DB already stores forecast/actual/previous for 6 countries' releases — the index can be built entirely from data the bot already collects, no new sources. Research needed: window length, normalization, whether the signal survives on the tradeable pair set (USDZAR/USDTRY/AUDUSD/USDJPY), correlation with the carry book.
    • Month-end / London 4 PM fix rebalancing flows — Institutional portfolio rebalancing concentrates around the WM/Reuters 4 PM London fix, strongest at month-end: documented pre-fix drift and post-fix reversal (Melvin & Prins, JIMF 2015; Evans, RFS 2018; the 2013-2015 fix-manipulation scandal exists precisely because these flows are real). Calendar-driven and quantitative — same operational shape as the event bot (known timestamp, short holding window). Testable directly on existing 1-min Dukascopy data for majors (AUDUSD, USDJPY, USDCAD). Research needed: does the effect survive spreads at retail size, month-end vs every-day fix, equity-market-performance conditioning (rebalancing direction follows relative equity/bond moves).
    • Commodity terms-of-trade signal for commodity currencies — "Commodity currencies" (AUD, CAD, NZD, ZAR) co-move with their export baskets; commodity-price momentum leads the currency at short horizons (Chen & Rogoff, JIE 2003; Ferraro, Rogoff & Rossi, JIMF 2015 — daily oil→CAD predictability). Signal: daily commodity index momentum (gold/iron ore→AUD, oil→CAD, platinum/gold→ZAR) → directional tilt, days-to-weeks holding. Data free via FRED/stooq. Fundamental, not TA — it's a terms-of-trade cash-flow argument. Research needed: which basket per currency, horizon sweep, overlap with carry (ZAR appears in both books).
    • COT positioning extremes (contrarian) — CFTC Commitments of Traders reports leveraged-fund FX futures positioning weekly (free). Extreme net positioning (e.g. >90th percentile of trailing 3y) marks crowded trades that unwind violently — useful either as a contrarian entry or, more conservatively, as a risk filter on the carry book (crowded carry = unwind risk, cf. Brunnermeier, Nagel & Pedersen, NBER 2008 "Carry Trades and Currency Crashes"). Evidence for standalone timing is mixed; evidence as a crash-risk conditioner is stronger. Cheap to test: weekly data, simple percentile logic. Note: no TRY/ZAR futures depth — applies to majors and MXN only.
    • FX volatility risk premium (longer-term) — Implied vol systematically exceeds realized vol in FX; harvesting it (short straddles/strangles on majors, delta-hedged) earns a documented premium (Londono & Zhou, Fed Board; Della Corte, Ramadorai & Sarno, JFE 2016). Genuinely diversifying vs the event straddle (which is effectively long event vol — a VRP book would be short vol on quiet days, complementary by construction). High effort: needs IB FX options (IDEALPRO options exist but liquidity/permissions must be verified), Greeks/hedging infrastructure, and margin far beyond the current account. Park until account size supports it; revisit alongside the live-trading transition.
    • Model drift detection — PARTIALLY COVERED: the weekly report (src/forex_bot/reporting/weekly.py) already compares live avg pips/trade against the Monte Carlo CIs and flags BELOW CI / ABOVE CI / IN RANGE per pair, and AnomalyDetector (src/forex_bot/reporting/alerts.py) fires on high slippage (>2× historical) and 3-loss streaks; the trade-review skill does live-vs-backtest comparison. Remaining for a "full" system: CIs are hardcoded for only USDZAR/USDTRY (not data-driven or all-pairs), and there is no dedicated regime/drift module or automated param-refresh trigger.
    • ~~FOMC-specific parameter split~~ — DONE: All three event types (NFP, CPI, FOMC) are independently profitable for both active pairs. FOMC optimal TP is slightly tighter (55-65 vs 70 pips) due to press conference reversal risk, but the marginal improvement doesn't justify splitting params yet. All 6 walk-forwards pass. See Event-Type Split Analysis.
    • ~~Automatic event data download~~ — DONE: Nightly cron job at 04:00 UTC (11 PM ET) runs download_dukascopy.py --skip-existing --timeframe 1min via asyncio.create_subprocess_exec. Appends new event data to existing CSVs automatically. See src/forex_bot/data/dukascopy.py.
    • ~~Remove Telegram alerts for IB connect/disconnect~~ — DONE: Demoted to log-only. Daily TWS restart no longer sends Telegram alerts.
    • ~~Spread/slippage logging and modeling~~ — DONE: entry_spread_pips, fill_price, slippage_pips tracked on every order. Spread captured at submission, slippage calculated on IB fill. Performance dashboard shows avg spread, avg slippage, total slippage. Auto-migrates existing SQLite DB.
      • How to use this data:
        • Validate MC assumptions: The Monte Carlo sims use fixed 50/70/10 pip params — if real slippage averages +3 pips, actual R:R is worse than modeled. Adjust MC sim to include avg slippage as a cost.
        • Per-pair spread profiles: Compare avg entry spreads for USDZAR vs USDTRY vs USDJPY. If one pair consistently has wider spreads at event time, tighten its straddle distance or widen the spread limit.
        • Event-type spread analysis: Do FOMC events have wider spreads than NFP? If so, FOMC may need different straddle params (the event-type split analysis already hinted at this).
        • Time-of-day patterns: BOJ events (3 AM UTC) may have different liquidity profiles than US 8:30 AM ET events. Spread data confirms or refutes this.
        • Slippage budget: Once you have 50+ fills, calculate the 95th percentile slippage. If it exceeds 5 pips, the straddle SL of 10 pips is getting eaten — consider widening SL or tightening straddle distance.
        • Live vs paper comparison: When switching to live, compare slippage distributions. Paper fills are instant and perfect; live fills have real market impact. This data quantifies the difference.

    Live Trading Readiness¶

    2FA Testing Required Before Going Live¶

    Paper trading does not require 2FA, so the IBC auto-start pipeline is fully unattended. Live trading will require 2FA via the IBKR Mobile app. Before switching to live:

    1. Test the IBC auto-start with live credentials and confirm the 2FA push notification arrives
    2. Verify 2FA-timeout relogin re-triggers the push if missed — ReloginAfterSecondFactorAuthenticationTimeout=yes is already set in ~/ibc/config.ini (alongside SecondFactorAuthenticationTimeout=180); note the earlier TWOFA_TIMEOUT_ACTION=restart reference was not a real IBC config key
    3. Confirm ReloginAfterSecondFactorAuthenticationTimeout=yes works as expected
    4. Time the full login flow (TWS launch > 2FA approval > API socket ready) to ensure the 1-hour pre-market window is sufficient
    5. Consider whether the nightly TWS restart also requires 2FA re-authentication on live accounts
    6. Update the cron schedule if the live login flow takes significantly longer than paper
